# Assumptions and Limits

## 前提条件

### 対象範囲
- 川崎競馬場のレースのみ。他場のレースは一切扱わない
- ダートのみ（川崎競馬場に芝コースは存在しない）
- 1日最大12レース、同時開催レースは存在しない

### 購入条件
- 最小購入単位: 100円
- コストモデル: なし（手数料・税金は計算に含めない）
- 賭式: Phase 1 では未定。Core は勝率・複勝圏率を出力し、賭式は後段で決定

### データ前提
- 過去走データは最低5走分あることが望ましいが、2走以上で予測実行可能
- 初出走馬はデータ不足として Gate で no-bet に分類される可能性が高い
- 馬体重は当日パドック後に確定するため、予測時点で未取得の場合がある

### 映像系前提
- 公式ライブ配信またはアーカイブ配信を人間が視聴する
- 映像の自動解析（画像認識AI等）は Phase 1 のスコープ外
- 観測結果は人間がタグ・コメント形式で入力する
- 映像ファイルは保存しない。保存するのはタグ・スコア・コメント・理由コードのみ

---

## 設計上の制約

### リーケージ防止
- 時系列分割のみ。ランダム分割は一切禁止
- 確定オッズ・確定着順・走破タイムは予測時特徴量に使用禁止
- 過去走データは対象レースの race_date より前の日付のデータのみ使用可能
- 解説者コメント・番組情報は主判定に使用禁止（参考表示のみ）

### 判定原則
- 映像からの判断は「見たままの観測事実」のみ
- 主観表現の禁止:「強そう」「調子が良さそう」等は不可
- 能力断定の禁止:「この馬は強い」等は不可
- 未来情報の参照禁止
- popularity baseline（人気順そのままの予測）を常に用意し、モデルが上回ることを確認する

### no-bet 中心設計
- デフォルトは no-bet。条件を満たした場合のみ bet に遷移する
- no-bet にも bet にも理由コードを付与する
- 映像系サブシステムには拒否権がある（Video Veto / Paddock Alert / Warmup Alert）
- 監視システムがROI閾値を下回った場合、強制停止（全レース no-bet）

### 映像系の独立性
- Race Video / Paddock / Warmup は Core に依存しない独立システム
- Core の予測結果を知らずに映像観測を行う（バイアス防止）
- 統合は Integration Layer でのみ行う
- 映像系は off / advisory / integrated の3モード

## 既知の限界

1. **データ量**: 川崎競馬は年間約160日開催、1日最大12レース。年間約1,920レース程度。統計的に十分なサンプルを得るには複数年のデータが必要

2. **馬場バイアス**: 川崎はダートのみで、雨天時の馬場変化が大きい。馬場状態の4段階分類では捉えきれない微妙な差がある

3. **クラス変動**: 地方競馬は中央からの転入・転出が多く、クラスの実力幅が大きい

4. **映像観測の主観性**: 人間が入力するため、観測者間・観測時点間のバラつきが存在する。タグ定義を厳密にしても完全には排除できない

5. **市場効率**: 川崎競馬のオッズは中央と比べてプール金額が小さく、大口投票による歪みが発生しやすい。これはリスクでもあり機会でもある

6. **小頭数問題**: 地方競馬は少頭数（4-6頭立て）のレースが存在し、予測の価値が限定的

7. **騎手・調教師の影響**: 地方競馬は所属騎手の偏りが大きく、特定の騎手のデータに引きずられやすい

---

## スコープ外（明示的に除外するもの）

- 他場の競馬（大井・船橋・浦和・中央等）
- 自動的な映像解析・画像認識
- リアルタイムオッズ取得・自動投票
- 複数レース横断のポートフォリオ最適化
- 非公式データソースの使用
- 映像ファイルの保存・管理
